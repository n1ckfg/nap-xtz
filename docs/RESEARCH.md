# NAP-XTZ & Tezos Contracts Research

## 1. Summary of Current Tezos Integration vs. Teia

### Current Implementation (`fa2-naplps.mligo` & `tezos.js`)
The `nap-xtz` repository currently relies on a minimal FA2-compatible NFT contract (`fa2-naplps.mligo`). 
- **Storage**: It uses standard FA2 big maps (`ledger`, `operators`, `token_metadata`).
- **Minting**: Anyone can call the `mint` entrypoint. The transaction expects a `Pair` containing the token's metadata map and the destination address (`to_`).
- **On-chain Data**: The NAPLPS vector graphics are converted to hex and stored directly in the `token_metadata` map under the key `"naplps"`. There is no IPFS or off-chain metadata involved.
- **Reading**: `tezos.js` reads the on-chain data by querying the TzKT API for the latest `token_metadata` big map entry and decoding the `"naplps"` hex string back to binary.

### Teia Contract
The Teia contract is a robust, production-ready extension of the FA2 standard.
- **Storage**: In addition to standard FA2 maps, it introduces `token_royalties` (for creator and minter royalty splits) and `token_data` (a specialized map for storing additional on-chain assets independently from TZIP-12 metadata).
- **Minting**: The `mint` entrypoint is restricted and can only be called by the `administrator` (typically a minter proxy or marketplace contract). It handles setting the initial supply, royalties, `token_metadata`, and `token_data`.
- **On-chain Views**: Implements `sp.onchain_view` methods (e.g., `token_data`, `token_royalties`) for fast on-chain reads by other contracts.

### Similarities & Differences
**Similarities:**
- Both are built on the FA2 (TZIP-12) standard.
- Both track balances via a `ledger` big map and support operator updates via `operators`.

**Differences:**
- **Access Control:** `nap-xtz` allows open minting directly to the token contract. Teia restricts minting to an admin (requiring users to interact via a marketplace proxy).
- **Metadata Structure:** `nap-xtz` overloads the standard `token_metadata` to hold the heavy NAPLPS payload. Teia segregates standard metadata (usually IPFS URIs) from heavy on-chain payloads, reserving `token_data` for the latter.
- **Royalties:** `nap-xtz` does not natively support royalties. Teia enforces complex royalty splits directly on-chain.

## 2. Adapting NAPLPS Publishing for Teia

To make the NAPLPS publishing methods in `tezos.js` compatible with the Teia ecosystem, the following adaptations are required:

### 2.1 Migrate Payload to `token_data`
Instead of injecting the NAPLPS hex payload into `token_metadata` (which Teia expects to strictly adhere to TZIP-12 standards like IPFS URIs), the payload should be placed in the `token_data` big map.
- **Action**: Encode the NAPLPS string to hex and format it to be injected into the `token_data` parameter during minting instead of `token_metadata`.

### 2.2 Update the Mint Transaction Structure
The Michelson parameter generated by `mintNaplpsToken` must be updated to match the Teia mint entrypoint (or its associated proxy contract).
- **Action**: Update the `mintValue` JSON in `tezos.js`. It will need to include fields for:
  - `token_metadata`: Standard TZIP-12 JSON (likely uploaded to IPFS first, or embedded as a `data:` URI).
  - `token_data`: The hex-encoded NAPLPS graphic.
  - `royalties`: A defined royalty split (e.g., 100 per mille / 10%).
  - `amount`: Initial supply.

### 2.3 Interact with the Correct Proxy Contract
Because Teia's core token contract restricts `mint` to the administrator, the DApp cannot mint directly to it.
- **Action**: Change `CONTRACT_ADDRESS` to point to the Teia Minter/Marketplace Proxy contract. The entrypoint name and parameter structure must align with the proxy's expected interface, which then forwards the request to the main FA2 contract.

### 2.4 Update Loading and Indexing Logic
`loadLatestToken()` currently fetches from the `token_metadata` big map of the `nap-xtz` contract.
- **Action**: Modify `readLatestNaplps()` to query the `token_data` big map of the Teia token contract. If using TzKT API, the endpoint will shift from `/bigmaps/token_metadata/...` to `/bigmaps/token_data/...` (or potentially utilize Teia's `sp.onchain_view` for `token_data` if invoking from another smart contract).
